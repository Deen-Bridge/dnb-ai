# Quran reference data — provenance

## `surah_index.json`

Generated by [`scripts/build_surah_index.py`](../../scripts/build_surah_index.py).

**Contents:** for each of the 114 surahs — number, transliterated name, Arabic
name, revelation place, ayah count, and common alternative spellings.

**Deliberately absent:** any Quranic text. This file is a *reference table*, not
a mushaf. Its only job is to let the service answer "is `2:300` a real ayah?"
offline, so an invalid reference is rejected before any network call and long
before a model has the chance to invent a verse.

**Numbering:** the Kufan count (6236 ayat in total), which is the numbering used
by the Uthmani mushaf, by Quran.com, and by every tafsir source this service
reads. The build script asserts both the surah count (114) and the total ayah
count (6236), so a typo in the table fails the build rather than shipping a
wrong bound.

**Licensing:** surah names and ayah counts are facts about the mushaf, not a
creative work, and carry no copyright.

## Ayah text and translations

Not bundled. Arabic text (`text_uthmani`) and translations are fetched at
request time from the [Quran.com API v4](https://api-docs.quran.com/) and cached
per ayah — see `tafsir.py`. Nothing is stored on disk.

## Tafsir text

Not bundled. Tafsir text is fetched per ayah from the same API, which serves the
works listed in `TAFSIR_REGISTRY` (`tafsir.py`). Attribution — the work's name,
author, and language — always comes from the API's own response for the resource
that was fetched; this service never labels a block of tafsir text from its own
memory of who wrote what.
